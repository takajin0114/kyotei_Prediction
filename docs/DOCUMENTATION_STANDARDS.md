# ドキュメント記載方法・方針

**最終更新日**: 2025-07-15  
**バージョン**: 1.0

---

## 📋 概要

このドキュメントは、kyotei_Predictionプロジェクトにおけるドキュメント記載方法・方針を定義します。一貫性のある、保守性の高いドキュメント作成を目指します。

---

## 🎯 ドキュメント構造の方針

### 1. **階層化された情報整理**

#### 最上位レベル
- **現在の状況サマリー**: 完了済み・進行中の全体像
- **優先度別タスクリスト**: 最優先・高優先・中優先・低優先の分類
- **自動化状況詳細**: システムの自動化レベル

#### 中位レベル
- **機能別実装タスク**: Phase別の詳細タスク
- **運用・保守ルール**: ドキュメント更新・運用方針
- **進捗・優先度**: 各機能の現在の状況

#### 詳細レベル
- **関連ドキュメント**: 詳細情報へのリンク
- **技術仕様**: 具体的な実装詳細
- **運用ガイド**: 実際の運用方法

### 2. **視覚的アイコンによる状態管理**

| アイコン | 意味 | 使用場面 |
|---------|------|----------|
| ✅ | 完了済み | 実装完了、テスト完了、運用開始 |
| 🔄 | 進行中 | 開発中、テスト中、運用中 |
| 📋 | 計画中 | 設計中、検討中、準備中 |
| 🎯 | 現在の状況 | 全体サマリー、現状把握 |
| 🚦 | 優先度順 | タスクリスト、優先度管理 |
| 📊 | 自動化状況 | システム状況、自動化レベル |
| 📝 | ドキュメント運用 | 更新ルール、運用方針 |
| 📈 | 進捗・優先度 | 各機能の進捗状況 |
| 📚 | 関連ドキュメント | 詳細情報へのリンク |

### 3. **時系列・バージョン管理**

#### バージョン管理
- **形式**: `X.Y`（例：3.4）
- **更新タイミング**: 重要な変更時
- **履歴**: 変更内容を明記

#### 日付管理
- **最終更新日**: 必ず明記
- **期限設定**: 具体的な日付（例：2025-07-20）
- **フェーズ期限**: 段階的な期限設定

## 📝 記載ルール・方針

### 1. **情報の重複を避ける**

#### 原則
- 同じ内容を複数箇所に記載しない
- 概要は簡潔に、詳細は別セクションに配置
- 関連ドキュメントへのリンクで詳細を参照

#### 実装方法
```markdown
## 概要
- 機能Aの実装完了
- → 詳細は「関連ドキュメント」セクション参照

## 関連ドキュメント
- [機能A詳細仕様](A_SPEC.md) - 実装詳細・技術仕様
```

### 2. **優先度の明確化**

#### 優先度分類
- **最優先（今週中）**: 即座に着手すべきタスク
- **高優先（来週）**: 短期間で完了すべきタスク
- **中優先（1-2ヶ月）**: 中期計画
- **低優先（3-6ヶ月）**: 長期計画

#### 記載例
```markdown
### 最優先（今週中）
1. **タスクA（期限: 2025-07-20）**
2. **タスクB（期限: 2025-07-22）**

### 高優先（来週）
3. **タスクC（期限: 2025-07-28）**
```

### 3. **進捗管理の統一**

#### 進捗記号
- `[ ]` = 未着手
- `[x]` = 完了
- `[~]` = 実装中

#### 記載例
```markdown
### Phase 1: 基本機能（MUST）
- [x] 機能Aの実装
- [~] 機能Bの実装
- [ ] 機能Cの実装
```

### 4. **期限管理**

#### 期限設定の原則
- 具体的な日付を設定
- フェーズ別の期限設定
- 依存関係を考慮した順序付け

#### 記載例
```markdown
### Phase 1: 基本UI・データ表示（期限: 2025-07-20）
### Phase 2: オッズデータ連携（期限: 2025-07-22）
### Phase 3: システム状況・管理（期限: 2025-07-24）
```

## 🔄 更新・運用方針

### 1. **定期更新ルール**

#### 更新頻度
- **週次**: 進捗・TODOの見直し
- **機能追加時**: 必ず更新
- **運用変更時**: 必ず更新

#### 更新内容
- 完了したタスクの移動
- 進行中タスクの状況更新
- 新規タスクの追加
- 期限の調整

### 2. **情報の整理・統合**

#### 完了タスクの処理
- 「完了済み」セクションに移動
- 実装詳細を関連ドキュメントに記載
- メインリストから削除

#### 進行中タスクの管理
- 「進行中」セクションで一元管理
- 進捗状況の定期的な更新
- ブロッカーの明記

### 3. **ドキュメント間の連携**

#### リンク管理
- 詳細は別ファイルに記載
- 適切なリンクで参照
- リンク切れの定期チェック

#### 重複回避
- 同じ内容の重複記載を避ける
- 参照リンクで詳細を提供
- メンテナンス性を重視

## 📚 テンプレート・例

### 1. **タスクリストテンプレート**

```markdown
## 📋 機能名 実装タスク詳細

### Phase 1: 基本機能（MUST）
- [ ] 機能Aの実装（期限: YYYY-MM-DD）
- [ ] 機能Bの実装（期限: YYYY-MM-DD）

### Phase 2: 拡張機能（WANT）
- [ ] 機能Cの実装（期限: YYYY-MM-DD）

#### 優先度・依存関係
- Phase 1 → Phase 2 の順で推奨

#### 進捗管理
- [ ] = 未着手
- [x] = 完了
- [~] = 実装中
```

### 2. **進捗サマリーテンプレート**

```markdown
## 📈 進捗・優先度（YYYY-MM-DD時点）

- **機能A**: ✅ 完了、詳細説明
- **機能B**: 🔄 進行中、進捗状況
- **機能C**: 📋 計画中、準備状況
```

### 3. **関連ドキュメントテンプレート**

```markdown
## 📚 関連ドキュメント

### 新規追加
- [新規ドキュメント](PATH.md) - 説明

### 既存ドキュメント
- [既存ドキュメント](PATH.md) - 説明
```
