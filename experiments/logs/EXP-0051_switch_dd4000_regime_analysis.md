# Experiment: EXP-0051 switch_dd4000 期間安定性・regime 差分析

## experiment id

EXP-0051

## purpose

EXP-0050 で実運用標準候補となった switch_dd4000 について、n_windows=36 を主対象に「どの期間で強く・どの期間で弱いか」を把握し、ブロック別 profit/ROI/DD・switch 発動回数・発動 window を集計する。実運用での注意点を明文化する。

## DB 存在確認結果と使用パス

- **確認したパス**: `kyotei_predictor/data/kyotei_races.sqlite`（リポジトリルート基準）。
- **実在確認**: 起動時に `_resolve_db_path()` で解決し `exists()` を確認。ローカルで実在を確認済み。
- **使用パス**: 未指定時は `_REPO_ROOT / "kyotei_predictor" / "data" / "kyotei_races.sqlite"`。本実行ではデフォルトを使用。

## 評価ロジック

- 選抜: d_hi475 固定（skip_top20pct + 4.30≤EV<4.75 + prob≥0.05）。
- 比較: normal_only, switch_dd4000, 参考で conservative_only。
- n_windows=36。ブロックは 6 windows × 6 blocks。各ブロックで block_profit / block_roi / block_drawdown / block_bet_count / block_hit_count を集計。
- switch_dd4000 の「累積DD≥4000 で当該 window を stake=80」の発動回数・発動 window 一覧を集計。

## 実装内容

- ツール: `kyotei_predictor/tools/run_exp0051_switch_dd4000_regime_analysis.py`
- EXP-0050 と同様に日別データを読み、ref_profit（normal 基準）から switch スケジュールを算出。窓単位の stake/payout/bet_count/hit_count を保持し、ブロック別に集計。
- 出力: 全体サマリ・ブロック別メトリクス・switch 発動回数・発動 window 一覧の JSON とコンソール表。

## regime 分析

1. **強い期間**: ブロック 0（1月末〜3月中旬）、ブロック 2（4月末〜6月初）で黒字かつ switch が normal より利益・ROI とも良い。前半 18 windows の early_half_profit は switch 14,928・normal 12,970。
2. **弱い期間**: ブロック 1, 3 は小幅赤字。ブロック 4, 5（7月後半〜10月）は大幅赤字（block_profit -6,948 / -4,312）。late_half_profit は switch -14,444・normal -17,460 で、後半全体で悪化。
3. **悪化局面の共通特徴**:
   - **後半（7月後半以降）で成績が悪化**: ブロック 4・5 で hit 数は 12、ROI は -36%〜-48%。bet_count は 146〜181 で、前半と比べて極端に少ないわけではない。
   - **DD 切替の効き方**: 後半では発動がほぼ全 window（w22–w35）に入り、stake=80 で損失を抑制。ブロック 4・5 でも switch の方が normal より損失・DD が小さい。
   - 特定月だけ極端に悪いというより、「後半 18 windows 全体がマイナスで、特にブロック 4・5 が大きい」という構造。
4. **switch 発動の偏り**: 22/36 と過半数で発動。前半は累積 DD が 4000 に達した窓で散発、後半は累積が一度 4000 を超えたあとほぼ継続して 80 が適用されている。

## 解釈

- switch_dd4000 は 36 windows 全体で唯一黒字（484）であり、全ブロックで normal 以上を維持。長期では「どの程度安定か」は、前半は安定してプラス・後半は大きくマイナスだが switch がそれを緩和する、という形で確認された。
- 「どういう局面で崩れるか」は、**7月後半〜10月（ブロック 4・5）の期間**で利益が大きく崩れる。この期間は switch にしても黒字にはならないが、normal より損失・max_drawdown を小さくできている。
- 実運用では、後半期間の成績悪化を前提に資金・モニタリングを設計する必要がある。

## 採用判断

- **結論**: **2) 標準採用だが注意条件を付与**とする。
- switch_dd4000 はそのまま実運用標準候補として採用する。ただし以下を注意条件として明文化する。
  - **注意条件**: 分析期間（36 windows）では、**後半（おおむね 7月後半以降に相当する test 期間）で成績が大きく悪化する傾向**がある。実運用では (1) 後半期間の損失を前提とした資金配分、(2) ブロック単位・月単位での利益モニタリング、(3) 必要に応じて後半の stake 据え置きや追加の DD 閾値の検討を推奨する。
- **judgment**: **adopt with caveats**（注意条件付き標準採用）。結果 JSON: outputs/selection_verified/exp0051_switch_dd4000_regime_analysis_results.json。
